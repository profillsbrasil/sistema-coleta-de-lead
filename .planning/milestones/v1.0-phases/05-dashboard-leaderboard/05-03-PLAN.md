    Integrate leaderboard fetch into the sync engine per D-09 and Pitfall 3 from RESEARCH.md.

    **Add fetchLeaderboard function** after the `pullChanges` function in `apps/web/src/lib/sync/engine.ts`:

    ```typescript
    async function fetchLeaderboard(): Promise<void> {
      try {
        const result = await syncClient.leaderboard.getRanking.query();
        await db.leaderboardCache.clear();
        const entries = result.ranking.map((r, i) => ({
          userId: r.userId,
          name: r.name,
          totalLeads: r.totalLeads,
          score: r.score,
          rank: i + 1,
          lastSyncAt: result.serverTimestamp,
        }));
        if (entries.length > 0) {
          await db.leaderboardCache.bulkPut(entries);
        }
      } catch {
        // Leaderboard fetch failure must NOT affect lead sync (per Pitfall 3)
      }
    }
    ```

    **Update syncCycle** to call `fetchLeaderboard()` after `pullChanges()`:

    In the `syncCycle` function's try block, add `fetchLeaderboard()` call after `pullChanges()` and before the finally block. It must be inside its own try/catch (already handled internally by `fetchLeaderboard`), but place it after pullChanges succeeds:

    ```typescript
    export async function syncCycle(): Promise<void> {
      if (isSyncing) {
        return;
      }

      isSyncing = true;
      try {
        await pushChanges();
        try {
          await uploadPendingPhotos();
        } catch {
          // Photo upload failure should not break sync cycle
        }
        await pullChanges();
        await fetchLeaderboard();
      } catch (error: unknown) {
        if (isUnauthorizedError(error)) {
          return;
        }
        throw error;
      } finally {
        isSyncing = false;
      }
    }
    ```

    CRITICAL constraints:
    - `fetchLeaderboard` has its own try/catch -- a failure never propagates to break the sync cycle (per Pitfall 3)
    - It runs AFTER pullChanges succeeds -- leaderboard data should be as fresh as possible
    - It uses the existing `syncClient` tRPC vanilla client (already configured with credentials)
    - It clears then bulk-puts to avoid stale entries in leaderboardCache
    - Import of `db` already exists at top of engine.ts -- the `leaderboardCache` table is available via the version 2 schema
